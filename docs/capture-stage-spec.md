## 0. 본 문서의 범위

CSVS 루프 중 **Capture 단계**의 베타 범위, 디바이스, 사용자 여정,
진입점, 출처 정책을 정의한다.

## 1. 베타 디바이스 — 두 개로 확정

```
1. Chrome Extension (Desktop)
2. PWA Share Target (Mobile)
```

**근거**
- 지식근로자 캡처의 80% 이상이 두 환경에서 발생
- 데스크탑: 깊이 있는 정보(논문, 보고서, 긴 기사)
- 모바일: 빠른 정보(릴스, 트윗, 뉴스 헤드라인)
- 두 경로가 같은 캡처 API로 흘러들어가 검증 큐에서 통합

**Phase 1 (M6+) 검토 항목**
- iOS·Android 네이티브 앱
- 데스크탑 네이티브 앱
- 이메일 전달 캡처
- 음성 입력

## 2. Capture 핵심 원칙 — 출처 보존이 우선

> **출처와 시점이 확보되지 않는 캡처는 베타에서 받지 않는다.**

이유: Lucid의 가치 명제는 "검증된 지식 자산"이다. 검증의 핵심
구성 요소는 출처·시점·검증자 세 가지다. 이 중 출처와 시점이 불명확한
정보는 그 자체로 검증 인프라의 신뢰성을 떨어뜨린다.

따라서 베타에서는 **출처 메타데이터(URL, 게시 시점, 작성자) 확보가
구조적으로 가능한 진입점만 허용**한다.

## 3. 베타 허용 진입점

### 3.1 Chrome Extension (Desktop)

| 진입점 | 출처 확보 | 베타 허용 | 비고 |
|--------|----------|----------|------|
| 페이지 전체 캡처 | ✅ URL + 메타데이터 자동 | ✅ | 기본 경로 |
| 하이라이트(선택 텍스트) | ✅ URL + 선택 위치 | ✅ | 부분 인용으로 기록 |
| 유튜브 영상 | ✅ URL + 채널 + 게시일 | ✅ | 자막 또는 Whisper |
| 페이지 내 이미지 우클릭 | ✅ URL + 이미지 소스 | ✅ | Vision 추출 |
| 크롬에서 열린 PDF | ✅ URL 또는 파일명 | ✅ | pdfplumber |
| **임의 파일 업로드** | ❌ 출처 입력 불명 | ❌ | Phase 1 |

### 3.2 PWA Share Target (Mobile)

| 진입점 | 출처 확보 | 베타 허용 | 비고 |
|--------|----------|----------|------|
| 공유 시트 (Share Target) | ✅ URL 자동 전달 | ✅ | 기본 경로 |
| URL 직접 붙여넣기 | ✅ URL 명시 | ✅ | iOS 우회용 |
| **스크린샷 업로드** | ❌ 출처 불명 | ❌ | 베타 제외 |
| **카메라 직접 촬영** | ❌ 출처 없음 | ❌ | 베타 제외 |
| **클립보드 텍스트 붙여넣기** | ⚠ 출처 불명 | ❌ | 베타 제외 |

### 3.3 베타에서 제외되는 진입점 — 처리 방침

스크린샷·카메라·임의 텍스트 입력 등 출처 확보가 어려운 캡처는
Phase 1 이후에 다음 정책 중 하나로 처리할 예정이다. 베타에서는
**기능 자체를 노출하지 않는다.**

```
정책 후보 (Phase 1에서 결정):
  A. 출처 입력을 사용자에게 강제 (텍스트 필드)
  B. AI가 텍스트 내용에서 출처 추정 → 사용자 확인
  C. "untraced" 메타데이터로 별도 큐에 분리, 검증 가중치 하향
  D. 받지 않음 (영구 제외)
```

이 결정은 별도 정책 문서에서 다룬다.

## 4. 사용자 여정 — Chrome Extension

### 4.1 페이지 전체 캡처 (기본 경로)

**진입 트리거**
사용자가 기사·논문을 읽다가 "저장할 만하다" 판단

**행동 흐름**

```
[1] 페이지 빈 공간에서 우클릭
       ↓
[2] 컨텍스트 메뉴 "Save to Lucid" 표시
       ↓
[3] 클릭
       ↓
[4] 우측 하단 오버레이 (300ms 슬라이드 인)
    ┌─────────────────────────────┐
    │ 📥 Capturing this page...   │
    │ ▓▓▓▓▓░░░░ 50%               │
    └─────────────────────────────┘
       ↓
[5] AI가 텍스트 추출 + 사실 분해 (5-15초)
       ↓
[6] 오버레이 변환
    ┌─────────────────────────────┐
    │ ✓ 3개 사실이 추출됐습니다     │
    │                              │
    │ 메모 (선택):                  │
    │ ┌─────────────────────────┐ │
    │ │ 왜 저장했는지...          │ │
    │ └─────────────────────────┘ │
    │                              │
    │ [지금 검증] [나중에]          │
    └─────────────────────────────┘
       ↓
[7] 분기
    A. "지금 검증" → Lucid 탭 열림 + HITL 화면
    B. "나중에" → PendingFact 큐 + 오버레이 닫힘 + 배지 +3
```

**완료 신호**
오버레이 닫힘 → 토스트 표시: "3 facts queued for review"

**핵심 마찰 측정**
- 클릭 수: 우클릭 1회 + 메뉴 클릭 1회 = 2회
- 메모 입력 강제 안 함
- "지금/나중에" 분기로 컨텍스트 스위칭 보호

### 4.2 하이라이트(선택 텍스트) 캡처

**진입 트리거**
긴 글에서 특정 문단·문장만 중요하다고 판단

**행동 흐름**

```
[1] 텍스트 드래그 선택
       ↓
[2] 선택 영역 상단에 미니 툴바
    ┌───────────────┐
    │ 📥 Lucid       │
    └───────────────┘
    또는 우클릭 → "Save selection to Lucid"
       ↓
[3] 오버레이 슬라이드 인
    - 선택 텍스트 미리보기
    - 페이지 URL 자동 첨부
       ↓
[4] AI 사실 분해 (1-3개)
       ↓
[5] 메모 + [지금 검증] / [나중에]
```

**페이지 전체 캡처와 차이점**
- 사실 추출 개수 적음(1-3개) → 그 자리 검증 가능성 높음
- **"지금 검증"을 기본 선택지로 강조 표시**

### 4.3 유튜브 영상 캡처

**진입 트리거**
강의·인터뷰·튜토리얼 시청 중 저장 결정

**행동 흐름**

```
[1] 영상 페이지에서 우클릭 또는 확장 아이콘 클릭
       ↓
[2] "Save YouTube video to Lucid"
       ↓
[3] 오버레이 미리보기
    ┌─────────────────────────────┐
    │ 📹 [영상 제목]                │
    │ 채널명 · 길이                 │
    │                              │
    │ ⏳ 자막 추출 중...            │
    └─────────────────────────────┘
       ↓
[4] 추출 우선순위
    1순위: youtube-transcript-api 자막
    2순위: yt-dlp + faster-whisper STT
       ↓
[5] 사실 분해 (5-20개)
       ↓
[6] 안내
    "20개 사실 추출됨. 검토하시겠습니까?"
    → 양 많으면 "나중에"가 기본 선택
```

**특이점**
- 사실 개수 많음 → 한 번에 검증 강요 금지
- PendingFact 큐 진입이 기본 동작
- 사용자는 시간 날 때 큐 비움

### 4.4 페이지 내 이미지 우클릭

```
[1] 이미지 위 우클릭
       ↓
[2] "Save image to Lucid"
       ↓
[3] 오버레이
    - 이미지 미리보기
    - 페이지 URL + 이미지 소스 URL 자동 첨부
       ↓
[4] Claude Vision으로 설명 + 사실 추출
       ↓
[5] 페이지 캡처와 동일한 흐름
```

### 4.5 PDF (Chrome에서 열린)

```
[1] 확장 아이콘 클릭
       ↓
[2] "Capture this PDF"
       ↓
[3] pdfplumber로 텍스트 추출
       ↓
[4] 페이지 캡처와 동일한 흐름
```

### 4.6 보조 UI 요소

**Chrome 툴바 확장 아이콘**

```
[L] ← Lucid 아이콘
     배지: PendingFact 개수
     클릭 시 팝업
     ┌─────────────────────────┐
     │ 📥 12 facts pending     │
     │ ✓ 247 validated         │
     │                          │
     │ [Review queue]           │
     │ [Open Lucid]             │
     └─────────────────────────┘
```

**키보드 단축키**

```
Cmd/Ctrl + Shift + L      → 현재 페이지 캡처
선택 후 Cmd/Ctrl + Shift + L → 하이라이트 캡처
```

## 5. 사용자 여정 — PWA Share Target

### 5.1 최초 설치 (1회)

```
[1] 사용자가 lucid.app 모바일 접속
       ↓
[2] "홈화면에 추가" 안내 (브라우저 프롬프트 또는 인앱 가이드)
       ↓
[3] 홈화면에 Lucid 아이콘 생성
       ↓
[4] OS가 Lucid를 공유 가능 앱으로 등록
```

### 5.2 공유 시트 캡처 (기본 경로)

```
[1] 인스타·트위터·유튜브·뉴스 앱에서 콘텐츠 보는 중
       ↓
[2] 공유 ↗ 버튼 탭
       ↓
[3] 공유 시트에서 "Lucid" 선택
       ↓
[4] Lucid PWA 열림 + URL/제목/이미지 자동 수신
       ↓
[5] 데스크탑 캡처와 동일한 오버레이 + 메모 + 검증/나중에 분기
```

**마찰 측정**
- 탭 수: 공유 버튼 1회 + Lucid 선택 1회 = 2회
- 인스타 저장 기능과 동등한 마찰

### 5.3 URL 직접 붙여넣기 (iOS 우회용)

```
[1] Lucid 홈화면 아이콘 탭
       ↓
[2] 메인 화면 입력창에 URL 붙여넣기
    ┌─────────────────────────────────┐
    │  Lucid                          │
    ├─────────────────────────────────┤
    │  ┌───────────────────────────┐ │
    │  │ Paste URL...              │ │
    │  └───────────────────────────┘ │
    │         [Capture]               │
    └─────────────────────────────────┘
       ↓
[3] 데스크탑 캡처와 동일한 흐름
```

**용도**
- iOS Share Target 한계 우회
- 카톡으로 받은 링크 처리
- 다른 앱이 공유 시트에 안 뜰 때

## 6. iOS vs Android 한계 인정

베타에서 받아들이는 한계:

```
Android
  - Share Target 매끄럽게 작동
  - 공유 시트에서 Lucid가 자연스럽게 등장
  - 백그라운드 처리 비교적 자유로움

iOS
  - PWA Share Target 지원 제한적
  - 공유 시트 등록이 Android만큼 매끄럽지 않음
  - 우회: URL 직접 붙여넣기, 또는 Safari 공유 시트의 "Lucid에서 열기"
  - 백그라운드 작업 제약
```

베타 사용자에게는 명시적으로 안내한다. 네이티브 앱은 PMF 확인 후
Phase 1에서 결정.

## 7. 모든 진입점이 같은 캡처 API로 흘러간다

데스크탑 5개 진입점 + 모바일 2개 진입점 = 총 7개 경로가 단일
백엔드 캡처 엔드포인트로 통합된다.

```
POST /api/spaces/{sid}/capture/{type}

type:
  - url      웹페이지·유튜브·기사
  - text     하이라이트(선택 텍스트)
  - file     PDF

Body:
  {
    source_url: str,           # 필수 — 출처 URL
    selected_text?: str,       # 하이라이트인 경우
    captured_at: ISO8601,      # 자동 기록
    user_agent: str,           # 디바이스 식별
    personal_note?: str,
    space_id: UUID
  }

Response:
  {
    pending_facts: [PendingFact],
    capture_id: UUID,
    source_metadata: { title, author, published_at }
  }
```

이 통합 구조 덕분에 Structure/Validate/Surface 단계는 진입점을
신경 쓸 필요 없다.

## 8. 데이터 여정 — Capture 단계

```
[입력]
  진입점 5+2 종에서 들어온 원시 데이터
  - URL, 선택 텍스트, 파일 바이트
  - 시점, 디바이스, 사용자 ID

       ↓ 자동 처리

[Source Detection]
  source_type 결정 (youtube/web/pdf/image)
  detector.py에서 URL 패턴 매칭

       ↓

[Extractor Pipeline]
  YouTube → 자막 또는 Whisper STT
  Web → newspaper3k + readability
  PDF → pdfplumber
  Image → Claude Vision
       ↓
  → merged_text + source_metadata

       ↓

[저장 상태]
  PendingFact 큐로 진입
  status: "pending_structuring"

       ↓

[다음 단계로의 핸드오프]
  Structure 단계가 merged_text를 받음
  소스 메타데이터 보존:
    - source_url
    - source_title
    - author/publisher (가능 시)
    - published_at (가능 시)
    - captured_at (항상)
    - captured_by (validator_id)
```

## 9. 측정 지표 — 베타에서 추적

```
Capture 성공률
  - 시도 대비 성공적으로 PendingFact 큐에 진입한 비율
  - 디바이스별, 진입점별 분리

캡처 마찰 시간
  - 우클릭/공유부터 오버레이 표시까지의 시간
  - 목표: P95 < 3초

추출 품질
  - 평균 PendingFact 개수 per capture
  - 출처 메타데이터 완전성 비율

진입점 분포
  - 5+2 경로 중 어떤 게 가장 많이 쓰이는지
  - 사용자 페르소나별 차이
```

## 10. 베타에서 명시적으로 제외하는 것

```
❌ 임의 파일 업로드 (출처 메타데이터 없음)
❌ 스크린샷 업로드 (출처 추적 불가)
❌ 카메라 직접 촬영 (오프라인 출처 없음)
❌ 클립보드 자동 감지 (iOS 권한 + 출처 불명)
❌ 음성 메모 입력 (Phase 2+)
❌ 이메일 전달 캡처 (Phase 2+)
❌ 네이티브 모바일 앱 (Phase 1+)
❌ 데스크탑 네이티브 앱 (Phase 1+)
```

이 결정의 일관된 근거: **출처 확보가 구조적으로 불가능하거나
운영 복잡도가 베타에 부적합한 경로는 받지 않는다.**

## 11. 미해결 결정 — 별도 정책 문서에서 다룰 것

```
Q1. 베타 후 어떤 정책으로 untraced 캡처를 받을지
    (출처 입력 강제 / AI 추정 / 별도 큐 / 영구 제외)

Q2. 동일 출처에서 여러 번 캡처 시 중복 처리
    (URL 단위? 콘텐츠 해시? 시점 별 스냅샷?)

Q3. 캡처 후 원본이 사라진 경우 (404, 삭제 영상)
    (이미 추출된 사실은 유지? 출처 무효 플래그?)

Q4. 저작권·ToS 경계
    (페이월 콘텐츠, 비공개 영상, 로그인 필요 페이지)
```

이 네 가지는 베타 진행하며 데이터 수집 후 결정.
